# Hvalsneskirkja-for-Sandgerði Spot Swap

**Date:** 2026-05-03
**Author:** Martin Herchl + Claude

## Context

Sandgerði Shore (slug `sandgerdi-shore`) is being retired in favor of Hvalsneskirkja, an old stone church 5 km south on the Reykjanes peninsula. Both spots are coastal drive-up viewing locations on the same peninsula with similar totality conditions, but Hvalsneskirkja offers a more scenic, more isolated setting and a famous historical-poet association.

This is a routine spot swap: one row out, one row in, photo files updated, seed files updated, horizon_check recomputed.

### Long-form text

**Description:**
> A historic 1887 stone church on the open south coast of the Reykjanes peninsula, set in dramatic dunes and lava flats with completely unobstructed views to the south and west. The poet Hallgrímur Pétursson served as pastor here in the 1640s — his daughter Steinunn's tombstone stands inside. The exposed coastal setting offers a clean western ocean horizon — exactly the direction of the eclipse sun at totality. About 5 km south of Sandgerði Shore and 30 minutes from Keflavík Airport.

**Parking:** Small free parking area at the church. Paved access road via Route 44 from Sandgerði.

**Terrain:** Grassy churchyard with gravel paths. Open coastal flatlands with sand dunes and lava plain to the south and west — completely unobstructed western horizon.

## Touchpoints

### 1. Repo changes (single PR)

1. **Photo:** `raw-photos/hvalsneskirkja-hero.jpg` already exists (1440×800, 1.8:1). Run `node scripts/process-spot-photos.mjs` to generate `public/images/spots/hvalsneskirkja-hero.webp` (1200×667) + `-thumb.webp` (600×333).
2. **Delete retired spot's webps:** `public/images/spots/sandgerdi-shore-hero.webp` + `-thumb.webp`.
3. **`scripts/seed-viewing-spots-v2.sql`:** Replace the `sandgerdi` INSERT block (lines ~256–263) with a new `hvalsneskirkja` INSERT block. Same column layout.
4. **`scripts/seed-spot-photos.sql`:** Replace the `sandgerdi-shore` UPDATE line with a new `hvalsneskirkja` UPDATE line referencing `hvalsneskirkja-hero.webp`. **Use `TODO_CREDIT` / `TODO_LICENSE` placeholders** (same pattern as Sky Lagoon in PR #30) — `raw-photos/CREDITS.txt` doesn't list this photo, so the user fills in the real source before running on Supabase.
5. **`scripts/output/seed-horizon-checks.sql`:** This file is regenerated by `recompute-spot-horizons.mjs`. Leave it alone in the PR; the regeneration step (below) will produce the new content when the user runs it.

`raw-photos/sandgerdi-shore-hero.jpg` is gitignored — leaving it on disk costs nothing and preserves the source if you ever change your mind.

### 2. Production DB session (after merge)

Run on Supabase, in this order:

Three statements, in order. The exact INSERT and UPDATE come straight from the updated `seed-viewing-spots-v2.sql` and `seed-spot-photos.sql`; pull them from those files after the PR merges.

```sql
-- a) Insert the new spot first (so the recompute script's API call below sees it).
--    Copy the INSERT block from scripts/seed-viewing-spots-v2.sql.

-- b) Add the photo entry.
--    Copy the UPDATE line from scripts/seed-spot-photos.sql, then replace
--    TODO_CREDIT / TODO_LICENSE with the real values for this photo.

-- c) Delete the retired spot.
DELETE FROM viewing_spots WHERE slug = 'sandgerdi-shore';
```

### 3. Horizon check (option 2 from design — run precompute pipeline)

After the production rows are in place:

```bash
node scripts/recompute-spot-horizons.mjs
# Pulls spots from https://eclipsechase.is, computes horizon_check for each
# using the local DEM binary, outputs scripts/output/seed-horizon-checks.sql.
# DEM binary is present locally (855 MB at server/data/dem/west-iceland-30m.bin).
```

Then apply the regenerated `scripts/output/seed-horizon-checks.sql` to Supabase. This will:
- Update `hvalsneskirkja`'s `horizon_check` JSONB with real DEM-derived data
- Skip the now-deleted `sandgerdi-shore` (its UPDATE will match 0 rows — no-op)

Optionally commit the regenerated `scripts/output/seed-horizon-checks.sql` to the repo as a follow-up so future runs start from the correct baseline.
